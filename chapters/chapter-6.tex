\chapter{Proving PRF-Induced MAC Security Equationally}
\lhead[\fancyplain{}{\bfseries\thepage}]{\fancyplain{}{\bfseries\rightmark}}
This chapter demonstrates that existential unforgeability under chosen-message attacks for pseudorandom-function-based message authentication codes may be rigorously articulated and validated inside the $\lamBLL$ framework. In particular, we formalize the standard reduction from MAC forgery to pseudorandom function
indistinguishability by encoding the $\macexp$ experiment, the adversary, and the
MAC oracle as $\lamBLL$ terms, and by reconstructing the reduction outlined in
Section~\ref{subsec:mac-security} using the equational reasoning principles of the
calculus. This formalization demonstrates that $\lamBLL$ is sufficiently expressive
to internalize classical cryptographic security reductions while enforcing polynomial-time constraints through its graded type system.

We begin by recalling the MAC-forgery experiment $\macexp$ and its pseudocode,
reported in Figure~\ref{fig:macforge-recall}, and by specifying the types of the
components of the experiment as they appear as subterms in the $\lamBLL$
formalization. The concrete implementation of the terms modeling the key-generation
algorithm $Gen$ and the MAC oracle $Mac_k(\cdot)$ depends on the specific MAC
scheme under consideration. For this reason, these terms are left abstract and will be instantiated only after fixing a scheme $\Pi$. In the following
paragraphs, we make this instantiation explicit for the schemes $\Pi^F$ and
$\widetilde{\Pi}$ that are used in the proof of Theorem~\ref{theorem:mac-security}.

\begin{figure}[htpb]
    \centering
    \begin{align*}
    &\macforge(n): \\
    &\quad k \gets Gen(1^n) \\
    &\quad (m,t) \gets A^{Mac_k(\cdot)}(1^n) \\
    &\quad \mathbb{Q} \gets \{m \ | \ A \ \text{queries} \ Mac_k(m) \} \\
    &\quad \mathtt{return}\,\,(m \notin \mathbb{Q} \land \vrfy_k(m,t) = 1)
    \end{align*}
    \caption{Pseudocode for $\mathtt{MacForge}$ experiment}
    \label{fig:macforge-recall}
\end{figure}

We now describe how the components of this experiment are typed in $\lamBLL$.
The key-generation algorithm $Gen$ is modeled as a probabilistic computation that,
given the security parameter, produces a secret key whose length is polynomial in
$i$, where $i$ denotes the security parameter. The adversary is modeled as a
higher-order computation that receives oracle access to the MAC function and outputs
a candidate forgery together with its internal state, which records the oracle
queries performed. The MAC oracle itself is represented as a term whose use is
controlled by a graded modality, enforcing a polynomial bound on the number of
queries.

\begin{align*}
\vdash_v^\Theta &: \UU \multimap \SS[i] && \tag{Type for $Gen$} \\
\vdash_v^\Theta &: (\SS[i] \otimes \SS[i]) \multimap \BB && \tag{Type for $\vrfy$} \\
\vdash_c^\Theta &: \SS[i] \multimap \bang_{q}\bigl(\SS[i] \multimap \SS[i]\bigr) && \tag{Type for $Oracle$} \\
\vdash_c^\Theta &: 
 \multimap \bang_{q}\bigl(\SS[i] \multimap \SS[i]\bigr) \multimap \bigl((\SS[i] \otimes \SS[i]) \otimes \SS[r]\bigr) && \tag{Type for $Adv$}
% \vdash_c^\Theta &: 
% \bang_{q}\bigl(\SS[i] \multimap \SS[i]\bigr) \multimap \bigl((\SS[i] \otimes \SS[i]) \otimes \SS[r]\bigr) && \tag{Type for $Adv$}
\end{align*}

Here, $i$, $r$, and $q$ denote polynomials in the security parameter. The oracle type $\SS[i] \multimap \bang_{q}(\SS[i] \multimap \SS[i])$ models a MAC oracle which, once instantiated with a secret key, maps messages of length $i$ to tags of the same length. The graded modality $\bang_{q}$ enforces a polynomial bound on the number of oracle queries that the adversary may perform. 
The adversary $Adv$ takes as input the closure obtained by currying and encapsulating the key within the $Oracle$ term. Its output type $(\SS[i] \otimes \SS[i]) \otimes \SS[r]$ consists of a candidate message-tag pair together with a string encoding the adversary's internal state. This state captures the information accumulated during oracle interactions, such as the history of queried messages. The verification function $\vrfy$ checks the validity of a proposed tag by recomputing the MAC on the message component of type $\SS[i]$ and comparing it with the supplied tag. Overall, the typing of $Adv$ ensures that both the number of oracle queries and the size of the internal state are polynomially bounded by $q$ and $r$, respectively. This reflects the fact that the adversary operates within probabilistic polynomial time, as enforced by the typing discipline of $\lamBLL$. The verification function $\vrfy$ checks the validity of the proposed tag by recomputing the MAC on the message component of type $\SS[i]$ and comparing the result with the supplied tag. The adversary, represented by the term $Adv$, is given access to a MAC oracle of type $\bang_{q}(\SS[i] \multimap \SS[i])$. Based on its interaction with the oracle, it produces a candidate forgery $(m,t)$ together with its internal state. The length of this state is bounded by the polynomial $r$, a fact ensured by the typing of $\lamBLL$ reflecting that the adversary operates within probabilistic polynomial time.

To construct the set
\[
\mathbb{Q} \gets \{\, m \mid A \text{ queries } Mac_k(m) \,\}
\]
we introduce an auxiliary function that allows inspection of the adversary’s query history. This is captured by the predicate $\mathtt{contains}$, whose purpose is to determine whether a given message occurs in the adversary’s internal representation of queried messages. More generally, $\mathtt{contains}$ checks whether a bitstring $\SS[i]$ appears as a substring of a longer bitstring $\SS[r]$. In the security experiment, these strings are instantiated respectively by the message $m$ of an attempted forgery $(m,t)$ and by the adversary’s internal state returned by the term $Adv$, which encodes the sequence of previous oracle queries. The type of the predicate is given by
\begin{align*}
\vdash^\Theta_v\ \mathtt{contains} : (\SS[r] \otimes \SS[i]) \multimap \BB
\end{align*}
indicating that it takes the adversary state and a candidate message as input and returns a boolean value.

It is worth noting that the type derivations of the terms introduced so far are parameterized by a reference context $\Theta$, which varies depending on the cryptographic experiment under consideration. The reference context will differ between the model for the encryption scheme $\Pi^F$ and that for $\widetilde{\Pi}$.
In the former case, the context $\Theta$ is empty, whereas in the latter it contains a reference used to model the random function $f$.

We define the $\lamBLL$ term corresponding to the forgery experiment, shown in Figure~\ref{fig:macforge-term}. The experiment begins by generating a fresh key $k$ and constructing the corresponding oracle $o$. The adversary $Adv$ is then given access to $o$ and returns both an attempted forgery $(m,t)$ and its internal state $s$, which encodes the history of oracle queries. The pair $(m,t)$ is destructured to handle the message and the tag separately, while the adversary state is bound to the variable $Q$. The experiment first checks whether the verification procedure $\vrfy$ accepts the pair $(m,t)$. If verification succeeds, it remains to test whether the message $m$ appears in the adversary's query history $Q$ using the predicate $\mathtt{contains}$. The experiment returns $\true$ if $m$ was not previously queried, and $\false$ otherwise. If verification fails, the experiment immediately returns $\false$.

\begin{figure}[ht]
    \centering
    \begin{align*}
    &MacForge \triangleq \\
    &\quad \letin{k = Gen \star}{} \\
    &\quad \letin{o = Oracle\, k}{} \\
    &\quad \letin{\langle mt, s \rangle = Adv\, o}{} \\
    &\quad \letin{\langle m, t \rangle = mt}{} \\
    &\quad \letin{Q = s}{} \\
    &\quad \letin{b = \vrfy\, \langle m, t \rangle}{} \\
    &\quad \mathtt{if}\, b\, \mathtt{then}\, \mathtt{not}(\mathtt{contains}(Q, m)) \\
    &\quad \mathtt{else}\, \false
    \end{align*}
    \caption{$\lamBLL$ term for the $\mathtt{MacForge}$ experiment.}
    \label{fig:macforge-term}
\end{figure}
